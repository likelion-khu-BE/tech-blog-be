# 0001. 컨트롤러 레벨 유저 정보 주입

Date: 2026-04-13

## Status

Accepted

## Context

"지금 로그인한 유저가 누구야?"를 코드에서 알아내는 방식이 두 가지 있다.

**방식 A — 컨트롤러에서 꺼내서 넘기기**

```java
// 컨트롤러
public ResponseEntity<?> createEvent(@CurrentUser CustomUserDetails user, ...) {
    eventService.createEvent(user.userId(), request);
}

// 서비스 — userId가 파라미터에 보인다
public void createEvent(Long userId, CreateEventRequest request) { ... }
```

**방식 B — 서비스 안에서 직접 꺼내기**

```java
// 컨트롤러
public ResponseEntity<?> deleteEvent(@PathVariable Long eventId) {
    eventService.deleteEvent(eventId); // userId는 안 넘김
}

// 서비스 — 안에서 몰래 꺼낸다
public void deleteEvent(Long eventId) {
    Long userId = SecurityUtils.getCurrentUserId();
}
```

둘 다 잘 동작한다. 문제는 팀원마다 다르게 쓰면 나중에 코드 읽을 때 헷갈린다는 거다.

### A가 좋은 점

- **테스트가 쉬움**: 서비스 테스트할 때 SecurityContext 세팅 없이 `userId`만 넘기면 끝
- **보면 안다**: 메서드 시그니처만 봐도 "아 이거 유저 정보 필요하구나" 파악 가능
- **책임이 깔끔**: "인증 정보 추출"은 컨트롤러 일, 서비스는 비즈니스 로직에만 집중
- **포트 쓸 때 투명함**: 다른 팀이 우리 포트 호출할 때 뭘 넘겨야 하는지 시그니처에 다 보임
- **불필요한 요청 차단**: 컨트롤러에서 인증을 먼저 확인하니까 서비스/DB까지 안 감

### B가 좋은 점

- **변경이 편함**: 나중에 유저 정보가 새로 필요해지면 `SecurityUtils` 한 줄이면 끝. 컨트롤러, 서비스 시그니처, 테스트를 줄줄이 고칠 필요 없음
- **파라미터가 짧아짐**: 전달 체인이 줄어들어 코드가 간결

### 팀 논의 (신선우, 장찬욱, 김우진)

- **선우**: 요청 흐름 관점에서 컨트롤러에서 꺼내는 게 자연스럽다
- **찬욱**: 다른 팀이 포트 쓸 때 파라미터가 명시적인 게 더 명확하다. 요청 처리량 관점에서도 컨트롤러에서 먼저 걸러내는 게 유리
- **우진**: B의 변경 용이성은 인정하지만, 그런 경우는 SecurityUtils로 예외 처리하면 됨

## Decision

**컨트롤러에서 `@CurrentUser`로 유저 정보를 꺼내서 서비스에 명시적으로 전달한다.**

예외적으로, 기존 서비스 메서드에 유저 정보가 추가로 필요해지는 경우에 한해 `SecurityUtils`를 사용할 수 있다. 새 기능을 설계할 때는 항상 컨트롤러 주입이 기본이다.

## Consequences

**좋은 점**
- 팀 전체가 하나의 패턴 → 코드 리뷰할 때 의도 파악이 빠름
- 서비스 메서드가 뭘 필요로 하는지 시그니처에 다 드러남 → 재사용할 때 "어 이거 왜 안 되지?" 방지
- 테스트에서 SecurityContext 없이 `userId`만 넘기면 됨

**안 좋은 점**
- 서비스에 유저 정보가 나중에 필요해지면 컨트롤러 → 서비스 시그니처 → 테스트 줄줄이 수정
- 파라미터 전달 체인이 길어질 수 있음 (Controller → Service → 내부 메서드)
