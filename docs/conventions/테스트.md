# 테스트 컨벤션

## 테스트 종류

| 종류 | 클래스 이름 | 뭘 검증하나 |
|------|-------------|-------------|
| 통합 테스트 | `*IntegrationTest` | API 엔드포인트가 실제로 동작하는지 (DB 포함) |
| 단위 테스트 | `*Test` | 서비스/유틸의 로직이 맞는지 (DB 없이) |
| 아키텍처 테스트 | `ArchitectureTest` | 모듈 의존성, 네이밍 규칙 위반 여부 |

## 테스트 구조 — @Nested로 기능별 묶기

```java
@SpringBootTest
@AutoConfigureMockMvc
class PostIntegrationTest {

    @Nested
    @DisplayName("글 생성")
    class CreatePost {

        @Test
        @DisplayName("정상 생성 → 201 Created")
        void createPost_success() throws Exception { ... }

        @Test
        @DisplayName("제목 누락 → 400 Bad Request")
        void createPost_noTitle() throws Exception { ... }
    }

    @Nested
    @DisplayName("글 조회")
    class GetPost {

        @Test
        @DisplayName("존재하는 글 → 200 OK")
        void getPost_success() throws Exception { ... }

        @Test
        @DisplayName("없는 글 → 404 Not Found")
        void getPost_notFound() throws Exception { ... }
    }
}
```

왜 `@Nested`? 테스트가 30개 넘어가면 평탄한 리스트로는 "어디서 뭘 테스트하는지" 찾기 어렵다. 기능별로 묶으면 IDE에서 트리 구조로 보인다.

## 메서드 네이밍

```
{대상}_{시나리오}
```

```java
void createPost_success()           // 정상 케이스
void createPost_noTitle()           // 제목 없이 요청
void login_wrongPassword()          // 비밀번호 틀림
void refresh_reuseDetection()       // 토큰 재사용 감지
```

- 한국어 메서드 이름은 쓰지 않는다 — 테스트 러너에서 깨질 수 있음
- 대신 `@DisplayName`에 한국어로 의도를 적는다

## @DisplayName은 결과를 적는다

```java
// O — "이렇게 하면 이렇게 된다"
@DisplayName("존재하지 않는 글 조회 → 404 Not Found")
@DisplayName("정상 가입 → PENDING 상태로 생성")

// X — 그냥 기능 이름만
@DisplayName("글 조회 테스트")
@DisplayName("회원가입")
```

## 테스트 데이터 정리

```java
@BeforeEach
void cleanUp() {
    // 각 테스트 전에 DB를 깨끗하게
    postRepository.deleteAll();
    userRepository.deleteAll();
}
```

왜 `@Transactional` 롤백을 안 쓰나? MockMvc로 요청을 보내면 별도 트랜잭션에서 실행된다. 테스트의 `@Transactional`로는 그 트랜잭션을 롤백할 수 없다. 그래서 `@BeforeEach`에서 수동 정리한다.
