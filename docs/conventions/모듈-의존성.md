## 디렉토리 구조

```
src/main/java/com/study/
├── StudyBeApplication.java        # 부트스트랩
│
├── auth/                          # Bounded Context (BC)
│   ├── domain/                    # 엔티티, 도메인 예외, 값객체
│   ├── application/               # 유스케이스 (Service)
│   ├── infrastructure/            # Repository, 외부 시스템 어댑터, 보안 인프라
│   └── presentation/              # Controller, Request/Response DTO
│
├── blog/
├── profile/
├── qna/
├── sessionboard/
│
└── shared/                        # 전역 공용 (어떤 BC도 모름)
    ├── config/                    # Spring 설정 (Security, CORS 등)
    ├── web/                       # 공용 Controller (Health 등)
    └── exception/                 # 글로벌 예외 핸들러
```

### Bounded Context 목록

- `auth` — 인증·계정
- `blog` — 게시글·댓글·관리자
- `profile` — 멤버·기수·팀·기술스택·활동
- `qna` — Q&A (placeholder, 비어있음)
- `sessionboard` — 세션·이벤트·자료
- `shared` — 전역 공용 (BC 아님)

## DDD 4계층

| 계층 | 책임 | 담는 것 |
|---|---|---|
| `domain` | 비즈니스 룰 | `@Entity`, Enum, 도메인 예외, 값객체 |
| `application` | 유스케이스 조율 | Service, DTO |
| `infrastructure` | 외부 시스템 연결 | Repository, JWT, 외부 API 클라이언트 |
| `presentation` | 외부와의 인터페이스 | Controller, Request/Response DTO |

### 계층 의존 방향 (단방향)

```
presentation ──┐
               ├──► application ──► domain
infrastructure ┘
```

- **`domain`은 어떤 외곽 계층도 모른다.** (DDD-Lite의 핵심 원칙)
  - JPA 어노테이션은 허용 (DDD-Lite — 진짜 헥사고날은 도메인 모델과 JPA 엔티티를 분리하지만 우리는 안 함)
  - 그러나 application/infrastructure/presentation 클래스 import 금지
- `application`은 `domain`만 알면 된다 (이상). 단, **Spring Data Repository 직접 inject은 허용** (Spring 표준 패턴, 스터디 실용성).
- `infrastructure`는 `domain`·`application`을 알 수 있다.
- `presentation`은 `application`·`domain`을 알 수 있다.

## Bounded Context 간 의존

> **현재**: 강제 룰 없음. 자유롭게 import 가능.

이유: 모놀리식 단일 jar라 격리의 실질 효과가 약함. 단일 트랜잭션·한 메모리·한 DI 컨테이너. 직접 의존이 자연스러움.

다만 **"의존성 그래프가 사이클을 만들지 않도록"** 자율적으로 신경 쓰기:
- `auth ← profile` (Member가 User FK 참조) — 정상
- `auth → profile` (User가 Member 참조)는 의도 없으면 피하기

추후 BC 격리 강도를 높여야 할 시점이 오면 그때 ArchUnit 등으로 강제. 지금은 **패키지 구조 + PR 리뷰 + 상식**으로 충분.

### 한 가지만 강제 (불문율)

- `shared`는 **어떤 BC도 import 안 함**. 모두에게 의존되는 공용 계층.

이건 룰이라기보단 자연스러운 구조 — `shared/config/SecurityConfig`가 `auth/application/AuthService`를 참조하면 의존 그래프가 꼬임.

## 새 BC 만들기 (예: qna 채워 넣을 때)

```bash
mkdir -p src/main/java/com/study/qna/{domain,application,infrastructure,presentation}
```

각 계층에 패키지 적절히 만들고, `domain/`부터 시작. 도메인 엔티티 → 서비스 → 리포지토리 → 컨트롤러 순서.

### 파일 위치 가이드

| 만드는 것 | 위치 |
|---|---|
| `@Entity` 클래스 | `qna/domain/` |
| Enum (도메인 의미) | `qna/domain/` |
| `*Service` | `qna/application/` |
| `*Repository` (Spring Data) | `qna/infrastructure/` |
| `*Controller` | `qna/presentation/` |
| Request/Response DTO (외부 노출) | `qna/presentation/dto/` |
| 내부 DTO (서비스 ↔ 컨트롤러) | `qna/application/dto/` |
| BC 전용 예외 | `qna/domain/exception/` |

## "Bounded Context란?"

도메인 주도 설계(DDD)에서 **하나의 일관된 모델이 통하는 경계**. 즉:
- `User`라는 단어가 `auth`와 `profile`에서 다른 의미일 수 있음 → 다른 BC
- 같은 BC 안에서는 한 단어가 한 가지 뜻만 가짐

우리 레포에서는 **팀 단위와 비슷하지만 정확히 일치 X**:
- `blog` BC = 블로그 팀의 도메인
- `profile` BC = 프로필 팀의 도메인
- 단, `auth`는 모든 팀이 공유하는 인증 도메인 (특정 팀 소유 X)

---

## 안 하는 것

- ❌ `domain`에 Spring `@Service`, `@Controller` 등 어노테이션
- ❌ `domain` 클래스가 `application`/`infrastructure`/`presentation` 클래스 import
- ❌ `shared`가 BC 패키지 import
- ❌ 같은 이름의 클래스를 여러 BC에 두고 `@Component` (Bean 이름 충돌)

## 옛 컨벤션과의 차이

| 옛 (멀티모듈) | 새 (싱글 모듈 + DDD-Lite) |
|---|---|
| `app/`, `common/`, `contract/`, 팀 모듈 | `src/` 하나만 |
| `BlogPort` 같은 Port 인터페이스 | 폐기 |
| ArchUnit으로 모듈 의존 강제 | 폐기 (학습 비용 > 가치) |
| 팀 모듈 ↔ 팀 모듈 직접 import 금지 | 자유 |
| 엔티티는 `common/entity/`에 | 각 BC의 `domain/`에 |
| Repository는 `common/repository/`에 | 각 BC의 `infrastructure/`에 |
