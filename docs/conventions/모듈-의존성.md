# 모듈 의존성 컨벤션

## 의존 구조

```
                    app
                     │
        ┌────────────┼────────────┐
        │            │            │
      blog         qna      session-board  profile
        │            │            │           │
        └────────────┼────────────┘           │
                     │                        │
                 contract              module-contract
                     │                        │
                     └────────┬───────────────┘
                              │
                           common
```

## 핵심 규칙

### 1. 팀 모듈끼리 직접 의존 금지

```java
// X — blog에서 qna를 직접 import
import com.study.qna.service.QnaService;  // 컴파일 에러 나야 정상

// O — contract의 Port를 통해 간접 의존
import com.study.contract.qna.QnaPort;
```

왜? blog가 qna를 직접 참조하면 qna를 고칠 때 blog가 깨진다. 4개 팀이 서로 직접 의존하면 누가 뭘 고치든 다른 팀 코드가 터진다. Port를 사이에 두면 "인터페이스만 맞으면" 내부 구현은 자유롭게 바꿀 수 있다.

이 규칙은 ArchUnit 테스트로 자동 검증된다 — 위반하면 CI에서 잡힌다.

### 2. Port 인터페이스는 contract 모듈에

다른 팀이 쓸 수 있는 기능은 contract에 `Port` 인터페이스로 정의한다.

```java
// contract 모듈 — 인터페이스만 정의
public interface BlogPort {
    List<BlogSummary> getRecentPosts(int limit);
}

// blog 모듈 — 구현
@Service
public class BlogService implements BlogPort {
    @Override
    public List<BlogSummary> getRecentPosts(int limit) { ... }
}
```

- 인터페이스 이름은 반드시 `*Port`로 끝난다 (ArchUnit으로 검증)
- 구현체는 팀 모듈 안에 숨긴다

### 3. common은 아래로만

common 모듈은 팀 모듈이나 contract를 의존하지 않는다.

common에 들어가는 것: 엔티티, 리포지토리, 보안(JWT, SecurityUtils), 공통 예외
common에 안 들어가는 것: 비즈니스 로직, 컨트롤러, 특정 팀의 DTO

### 4. app은 조립 담당

app 모듈은 모든 모듈을 의존하고, 실행 가능한 애플리케이션으로 조립한다.

app에 들어가는 것: SecurityConfig, GlobalExceptionHandler, 인증 컨트롤러, main 클래스
app에 안 들어가는 것: 팀별 비즈니스 로직 (그건 각 팀 모듈에)

## build.gradle에서 의존성 선언

```gradle
// 팀 모듈 (blog, qna, profile, session-board)
dependencies {
    implementation project(':common')
    implementation project(':contract')    // 다른 팀의 Port 사용
}

// contract
dependencies {
    implementation project(':common')      // 팀 모듈 의존 금지
}

// common
dependencies {
    // 다른 프로젝트 모듈 의존 금지
}
```
