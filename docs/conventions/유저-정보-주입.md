# 유저 정보 주입 컨벤션

> 왜 이렇게 결정했는지 궁금하면: [ADR-0001 컨트롤러 레벨 유저 정보 주입](../adr/0001-컨트롤러-레벨-유저정보-주입.md)

## 기본 패턴

컨트롤러에서 `@CurrentUser`로 유저 정보를 꺼내고, 서비스에는 `userId`를 넘긴다.

```java
// 컨트롤러
@PostMapping("/events")
public ResponseEntity<?> createEvent(
        @CurrentUser CustomUserDetails user,        // 여기서 꺼내고
        @RequestBody CreateEventRequest request) {
    eventService.createEvent(user.userId(), request); // userId만 넘긴다
    return ResponseEntity.ok().build();
}

// 서비스
public void createEvent(Long userId, CreateEventRequest request) {
    // userId를 파라미터로 받아서 쓴다
    // SecurityUtils 같은 거 호출하지 않는다
}
```

핵심: **서비스 메서드의 파라미터만 보면 "이 메서드가 유저 정보를 쓰는지 안 쓰는지" 바로 알 수 있어야 한다.**

## 이건 하면 안 됨

```java
// X — 새 기능인데 서비스에서 직접 꺼내는 거
public void createEvent(CreateEventRequest request) {
    Long userId = SecurityUtils.getCurrentUserId(); // 밖에서 보면 userId가 필요한지 모름
}
```

## 예외: SecurityUtils는 언제 써?

이미 만들어진 서비스 메서드에 유저 정보가 나중에 필요해진 경우.
시그니처 바꾸는 게 부담이 크면 `SecurityUtils.getCurrentUserId()`로 꺼내 쓸 수 있다.

**새 기능 설계할 때는 항상 컨트롤러 주입이 기본이다.** "편하니까 SecurityUtils 쓸게요"는 안 된다.
