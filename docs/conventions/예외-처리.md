# 예외 처리 컨벤션

## 구조

```
RuntimeException
  └── 도메인별 베이스 예외 (AuthException, BlogException, ...)
        ├── InvalidCredentialsException
        ├── UserNotActiveException
        └── ...
```

모든 커스텀 예외는 `RuntimeException`을 상속하는 **도메인별 베이스 클래스**를 거친다. `Exception`(checked)은 쓰지 않는다.

왜? checked exception은 호출하는 쪽이 전부 try-catch 또는 throws를 달아야 해서 코드가 지저분해진다. Spring도 런타임 예외를 기본으로 쓴다.

## 예외 클래스 만드는 법

```java
// 1. 도메인별 베이스 (common 모듈에 정의)
public abstract class BlogException extends RuntimeException {
    protected BlogException(String message) {
        super(message);
    }
}

// 2. 구체적 예외 (common 모듈에 정의)
public class PostNotFoundException extends BlogException {
    public PostNotFoundException(Long postId) {
        super("게시글을 찾을 수 없습니다: " + postId);
    }
}
```

- 메시지는 **한국어** — 프론트엔드가 사용자에게 그대로 보여줄 수 있게
- 생성자에서 메시지를 확정 — 던지는 쪽에서 매번 메시지를 조립하지 않는다

## GlobalExceptionHandler 등록

새 예외를 만들면 `GlobalExceptionHandler`에 핸들러를 추가한다.

```java
// app 모듈 — GlobalExceptionHandler.java
@ExceptionHandler(PostNotFoundException.class)
public ResponseEntity<Map<String, Object>> handlePostNotFound(PostNotFoundException e) {
    return errorResponse(HttpStatus.NOT_FOUND, e.getMessage());
}
```

핸들러를 안 추가하면? Spring 기본 500 에러가 나간다. 프론트엔드에서 에러 메시지를 파싱할 수 없게 됨.

## 예외 ↔ HTTP 상태 코드 매핑

| 상황 | HTTP 코드 | 예시 |
|------|-----------|------|
| 리소스 없음 | 404 | `PostNotFoundException` |
| 입력 값 잘못됨 | 400 | `@Valid` 검증 실패 |
| 인증 실패 | 401 | `InvalidCredentialsException` |
| 권한 없음 | 403 | `@PreAuthorize` 실패 |
| 중복 리소스 | 409 | `EmailAlreadyExistsException` |
| 비즈니스 규칙 위반 | 422 | `AlreadyApprovedException` |

## 이건 하면 안 됨

```java
// X — 서비스에서 직접 ResponseEntity를 만드는 거
public ResponseEntity<?> createPost(...) {
    if (duplicate) return ResponseEntity.status(409).body("중복");
}

// O — 예외를 던지고 GlobalExceptionHandler가 처리
public PostResponse createPost(...) {
    if (duplicate) throw new PostAlreadyExistsException(title);
}
```

왜? 서비스가 HTTP를 알면 안 된다. 서비스는 비즈니스 로직만, HTTP 매핑은 GlobalExceptionHandler가.
