## DB CASCADE 수정 내역

이번 실행에서 기존 "알려진 한계"로 표시됐던 FK 제약 위반 문제를 근본적으로 해결했습니다.

### 문제

`PostService.deletePost()`, `AdminService.forceDeletePost()`, `CommentService.deleteComment()`가  
의존 데이터(`post_likes`, `post_bookmarks`, `comments`, `comment_likes`, 대댓글)를 수동으로  
삭제하지 않아서, 의존 데이터가 있는 포스트/댓글 삭제 시 FK 제약 위반 예외 발생.

### 해결

DB 수준의 `ON DELETE CASCADE` 설정으로 해결:

1. **JPA 엔티티에 `@OnDelete(action = OnDeleteAction.CASCADE)` 추가**  
   Hibernate가 DDL 생성 시 FK에 `ON DELETE CASCADE`를 포함시킴  
   → H2 `create-drop` (테스트 환경)에도 CASCADE가 적용됨

   | 엔티티 | FK 필드 | CASCADE 효과 |
   |--------|---------|--------------|
   | `PostLike` | `post` | posts 삭제 → post_likes 자동 삭제 |
   | `PostBookmark` | `post` | posts 삭제 → post_bookmarks 자동 삭제 |
   | `PostTag` | `post` | posts 삭제 → post_tags 자동 삭제 |
   | `Comment` | `parent` | 루트 댓글 삭제 → 대댓글 자동 삭제 |
   | `CommentLike` | `comment` | comments 삭제 → comment_likes 자동 삭제 |

2. **Flyway 마이그레이션 (`blog/flyway/V1__init_schema.sql`) 생성**  
   프로덕션 DB에서도 동일한 CASCADE 제약을 보장

3. **서비스 코드 변경 없음**  
   DB CASCADE로 처리하므로 `deletePost()`, `deleteComment()` 구현 변경 불필요
